# コードレビュースキル

現在のブランチの変更を **批判的な視点** でレビューする。
良い点を褒めるのではなく、**問題点・改善点の発見** に集中すること。

## 手順

1. メインブランチ（`main`）との差分を取得する
2. 変更されたファイルの一覧と差分内容を確認する
3. 必要に応じて変更ファイルの周辺コード（変更されていない部分）も読み、文脈を理解する
4. 以下のレビュー観点すべてについて検査する
5. 結果をレポートとして出力する

## 差分の取得

以下のコマンドで差分を取得すること:

```bash
# 変更ファイル一覧
git diff main...HEAD --name-status

# 差分の詳細
git diff main...HEAD

# コミット履歴
git log main..HEAD --oneline
```

## レビュー観点

### 1. 設計・アーキテクチャ

- **責務の分離**: 一つのモジュール/関数が複数の責務を持っていないか
- **依存関係の方向**: レイヤー間の依存が正しいか（renderer → main への直接依存など）
- **Electron アーキテクチャ違反**: renderer から Node.js API を直接使っていないか、IPC を経由しているか
- **抽象化の適切さ**: 過剰な抽象化や、逆に抽象化が不足していないか
- **既存パターンとの一貫性**: プロジェクト内の既存コードと異なるパターンを不必要に導入していないか

### 2. コーディングスタイル・規約

- **TypeScript strict モード準拠**: `any` 型の使用、型安全性が損なわれていないか
- **`import type` の使用**: 型のみのインポートに `import type` を使っているか
- **インポートパスの拡張子**: ファイル拡張子付きのインポートパスを使っているか
- **命名規則**: 変数・関数・型の命名が適切か、プロジェクトの慣習に従っているか
- **AGENTS.md のルール違反**: `Bun.*` API の使用、`nodeIntegration: true` など禁止事項に抵触していないか

### 3. テスト

- **テストの有無**: 新規ロジックに対するテストが書かれているか
- **テストの質**: ハッピーパスだけでなく、エッジケースやエラーケースもカバーしているか
- **テストファイルの配置**: `*.test.ts` / `*.test.tsx` が対象ファイルと同階層にあるか
- **テストが不足している箇所の具体的な指摘**: 「どの関数/ロジックにテストが必要か」を明示する

### 4. エラーハンドリング

- **エラーの握りつぶし**: `catch` ブロックが空、または `console.log` だけで終わっていないか
- **ユーザーへのフィードバック**: エラー発生時にユーザーに適切に通知しているか
- **型安全なエラー処理**: unknown 型の catch、適切な型ガード

### 5. セキュリティ

- **機密情報の扱い**: アクセストークンがハードコードされていないか、`safeStorage` を使っているか
- **IPC の安全性**: `contextIsolation: true` が維持されているか、IPC チャンネルが型安全か
- **入力のサニタイズ**: 外部入力（API レスポンス含む）を信頼しすぎていないか
- **XSS リスク**: `dangerouslySetInnerHTML` の使用、innerHTML への直接代入

### 6. パフォーマンス

- **不要な再レンダリング**: React コンポーネントで不必要な再レンダリングが発生しないか
- **メモリリーク**: イベントリスナーの解除忘れ、useEffect のクリーンアップ漏れ
- **N+1 問題**: API 呼び出しがループ内で行われていないか

### 7. 可読性・保守性

- **複雑な条件分岐**: ネストが深すぎないか、早期 return で簡潔にできないか
- **マジックナンバー/マジックストリング**: 定数化すべき値がハードコードされていないか
- **コードの重複**: DRY 原則に反する重複がないか（ただし過剰な共通化は不要）

## 出力フォーマット

以下の形式でレビュー結果を出力する:

### サマリー

変更の概要を 1〜2 文で記述する。

### 指摘事項

各指摘を以下の形式で記述する。指摘がない観点はスキップしてよい。

#### 🔴 Critical（必ず修正すべき）

セキュリティ脆弱性、データ損失リスク、アーキテクチャ違反など。

- **[観点]** `ファイルパス:行番号` — 指摘内容。修正案があれば提示する。

#### 🟡 Warning（修正を推奨）

バグの可能性、テスト不足、エラーハンドリング不備など。

- **[観点]** `ファイルパス:行番号` — 指摘内容。

#### 🔵 Info（改善提案）

可読性向上、リファクタリング候補、ベストプラクティスの提案など。

- **[観点]** `ファイルパス:行番号` — 指摘内容。

### テストカバレッジ評価

テストが不足しているロジックを具体的にリストアップする。

### 総合評価

変更全体の品質を簡潔に評価する（1〜3 文）。
